# Pepper — Developer Guide

Pepper is a personal assistant built on Claude Code. This repo contains the source code.
The runtime workspace lives at `~/.pepper/` (generated by `pepper init`).

## Commands

```bash
just check          # Fast dev loop: lint, types, unit tests (~30s)
just gate           # Full pre-merge: all quality + all tests + coverage (~3 min)
just test           # All tests with parallel execution
just test-unit      # Unit tests only
just format         # Auto-fix formatting and lint issues
just install        # Install all dependencies
just hooks          # Install pre-commit hooks
uv run pepper --help             # CLI help
uv run pepper init               # Generate ~/.pepper/ runtime
uv run pepper init --migrate     # Migrate existing vault
```

## Architecture

```
src/pepper/
├── cli.py              # CLI entry point (pepper init, start, stop, status)
├── process.py          # PID file and process management
├── attachments.py      # Attachment download, storage, cleanup
├── init/               # Runtime workspace generator
│   ├── generator.py    # Creates ~/.pepper/ from templates
│   └── templates/      # Jinja2 templates for runtime config
├── hooks/              # Claude Code session hooks
│   ├── shared.py       # Vault I/O utilities
│   ├── session_start_context.py
│   ├── session_end_flush.py
│   └── pre_compact_flush.py
├── channel/            # Channel MCP server (message router)
│   ├── server.py       # Low-level MCP + HTTP server
│   └── router.py       # Routing table with TTL
├── skills/             # Claude Code skills (installed by pepper init)
└── integrations/
    └── discord/        # Discord bot + scheduler MCP server
tests/                  # All tests (flat structure, Phase 2 will reorganize)
```

## Entry Points (pyproject.toml)

- `pepper` -> `pepper.cli:app` — Main CLI
- `pepper-channel` -> `pepper.channel.server:main` — Channel MCP server
- `pepper-discord` -> `pepper.integrations.discord.mcp_server:run` — Discord MCP server

## Dev vs Runtime

| | This repo (dev) | ~/.pepper/ (runtime) |
|---|---|---|
| Purpose | Write code, run tests | Run Pepper as assistant |
| .claude/ | Dev settings only | Full config with hooks, MCP, skills |
| CLAUDE.md | This dev guide | Pepper persona + identity |
| Vault | Not present | ~/.pepper/Memory/ |

## Conventions

- Python 3.12 with strict MyPy typing
- `Any` at library boundaries only. Convert to typed Pydantic models immediately at the boundary. No `Any` in internal function signatures.
- Google-style docstrings on all public modules, classes, methods, and functions
- Feature branches, no direct commits to main
- Tests organized by type: `tests/unit/`, `tests/integration/`, `tests/e2e/` (Phase 2)
- Network is blocked by default in unit tests. Use `@pytest.mark.integration` or `@pytest.mark.e2e` to allow network access.

## Shipping

Always use `/ship` to create PRs. Never create PRs manually or through other skills.

## Gotchas

- The channel server uses the low-level MCP Server API (not FastMCP) for custom `notifications/claude/channel` support
- `pepper start` passes `--dangerously-load-development-channels server:pepper-channel` to Claude Code — required for channel notifications
- Discord integration imports use relative imports (`.config`, `.bot`, etc.) — they're a sub-package, not standalone scripts
- Hooks run as `python -m pepper.hooks.<name>` from the runtime directory — they find the vault via `PEPPER_VAULT_PATH` env var or default `~/.pepper/Memory`
- The `jobs.yaml` scheduler supports two job types: prompt-based (posts to channel server) and function-based (`type: function` calls Python directly)

## Skill routing

When the user's request matches an available skill, ALWAYS invoke it using the Skill
tool as your FIRST action. Do NOT answer directly, do NOT use other tools first.

Key routing rules:
- Bugs, errors, "why is this broken" -> invoke investigate
- Ship, deploy, push, create PR -> invoke ship
- QA, test the site, find bugs -> invoke qa
- Code review, check my diff -> invoke review
