# Pepper MVP: Channel Server + Discord Bot

## Goal

Give Pepper a nervous system. One channel server routes all messages in and out of Pepper's Claude Code session. Discord is the first integration — Jeff can talk to Pepper from his phone or desktop, and Pepper can reply, react, and show typing indicators.

## Architecture

```
Discord Bot (Python)          Email Poller (Python, future)       Heartbeat (Python, future)
     |                              |                                  |
     |  POST /message               |  POST /message                   |  POST /message
     v                              v                                  v
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                    Channel Server (TypeScript/Bun)                      │
  │                    localhost:8788 — MCP server                          │
  │                                                                         │
  │  Inbound:  POST /message → validate → push <channel> event to session  │
  │  Outbound: reply tool called → lookup source by chat_id → SSE stream   │
  │  Register: POST /register → store source + callback info               │
  │  Health:   GET /health → status + registered integrations              │
  │  Events:   GET /events?source=discord → filtered SSE stream            │
  └────────────────────────────┬────────────────────────────────────────────┘
                               │ MCP stdio
                               v
                    ┌──────────────────┐
                    │  Pepper Session   │
                    │  (Claude Code)    │
                    │  Interactive,     │
                    │  always-on        │
                    └──────────────────┘
```

**Inside Claude Code (TypeScript):** Channel server. Speaks MCP protocol over stdio, pushes events into Pepper's session, receives replies via the `reply` tool.

**Outside Claude Code (Python):** All integrations. Standalone processes that make HTTP calls to the channel server. Each is its own uv project.

## Channel Server

Evolves the POC at `channel/pepper-channel.ts` into a production message router.

### Inbound

`POST /message` accepts JSON:

```json
{
  "source": "discord",
  "chat_id": "guild-123-channel-456-msg-789",
  "sender": "jeff",
  "content": "What's on my calendar today?",
  "metadata": {
    "guild_id": "123",
    "channel_id": "456",
    "message_id": "789",
    "is_dm": false
  }
}
```

Validation:
- `source` and `content` are required
- `chat_id` is required (generated by the integration, not the channel server)
- `sender` defaults to "unknown" if omitted

The channel server:
1. Stores the `chat_id -> source` mapping in a routing table (in-memory Map)
2. Pushes a `notifications/claude/channel` event with `content` as the body and all other fields as `meta` attributes
3. Returns `{ "status": "queued", "chat_id": "..." }`

### Outbound (Reply Routing)

When Pepper calls the `reply` tool with `{ chat_id, text, metadata }`:
1. Look up `source` from the routing table by `chat_id`
2. Broadcast to the SSE stream for that source
3. Return `{ type: "text", text: "sent" }` to Pepper

The `metadata` field on replies is optional and can contain:
- `reactions`: array of emoji names to apply to the original message (e.g., `["thumbs_up", "eyes"]`)
- `type`: "message" (default) or "reaction" (reaction-only, no text reply)
- `embed`: object with embed fields (title, description, color, fields array)

### SSE Streams

`GET /events?source=discord` — filtered stream for a specific integration. Each event is a JSON line:

```json
{
  "chat_id": "guild-123-channel-456-msg-789",
  "text": "Let me check your calendar...",
  "metadata": { "embed": { "title": "Today's Calendar", "color": 3447003 } },
  "ts": "2026-04-06T12:00:00Z"
}
```

`GET /events` (no filter) — all replies, for debugging.

### Registration

`POST /register` accepts:

```json
{
  "source": "discord",
  "description": "Discord bot for Jeff's server"
}
```

Returns `{ "status": "registered" }`. Registration is informational — the channel server routes by the `source` field on inbound messages, not by pre-registration. But registration lets `/health` report which integrations are connected.

### Health

`GET /health` returns:

```json
{
  "status": "ok",
  "port": 8788,
  "registered_sources": ["discord"],
  "routing_table_size": 42,
  "uptime_seconds": 3600
}
```

### MCP Configuration

The channel server declares:
- `claude/channel` capability (required — registers notification listener)
- `tools` capability (required — exposes the reply tool)
- `instructions` string telling Pepper how messages arrive and how to reply

The `reply` tool schema:

```json
{
  "name": "reply",
  "inputSchema": {
    "type": "object",
    "properties": {
      "chat_id": { "type": "string", "description": "The conversation to reply in (from the channel tag)" },
      "text": { "type": "string", "description": "The message to send" },
      "metadata": {
        "type": "object",
        "description": "Optional: reactions, embed, type",
        "properties": {
          "reactions": { "type": "array", "items": { "type": "string" } },
          "type": { "type": "string", "enum": ["message", "reaction"] },
          "embed": { "type": "object" }
        }
      }
    },
    "required": ["chat_id"]
  }
}
```

### Routing Table Cleanup

Chat ID mappings are kept for 24 hours, then evicted. This prevents unbounded memory growth. The 24-hour window is configurable via `PEPPER_ROUTE_TTL_HOURS`.

## Discord Bot

Standalone Python process. Bridges Discord to the channel server over HTTP.

### Package

- Library: `discord.py` (v2.x, the maintained version)
- HTTP client: `httpx` (async, matches discord.py's async model)
- Config: `python-dotenv` for token management

### Intents

Required Discord intents:
- `guilds` — know which servers the bot is in
- `guild_messages` — receive messages in channels
- `dm_messages` — receive DMs
- `message_content` (privileged) — read message text (must be enabled in Developer Portal)

### Inbound (Discord -> Pepper)

On `on_message` event:
1. Ignore messages from the bot itself
2. Build the payload:
   - `source`: "discord"
   - `chat_id`: `f"{message.guild.id}-{message.channel.id}-{message.id}"` (or `f"dm-{message.channel.id}-{message.id}"` for DMs)
   - `sender`: `message.author.display_name`
   - `content`: `message.content`
   - `metadata`: `{ guild_id, channel_id, message_id, is_dm, author_id }`
3. Show typing indicator in the channel: `async with message.channel.typing()`
4. POST to `http://localhost:8788/message`

### Outbound (Pepper -> Discord)

The bot connects to `GET http://localhost:8788/events?source=discord` as an SSE client on startup. When a reply arrives:

1. Parse the `chat_id` to extract guild_id, channel_id, message_id
2. Fetch the target channel from Discord's cache
3. If `metadata.type` is "reaction": apply reactions to the original message, no text reply
4. If `metadata.reactions` is present: apply reactions to the original message
5. If `metadata.embed` is present: build a Discord Embed and send it
6. Otherwise: send `text` as a plain message (Discord renders markdown natively)

### Rich Embeds

Pepper can send structured replies using Discord embeds. The `metadata.embed` object maps to Discord's embed format:

```json
{
  "title": "Daily Briefing",
  "description": "Here's what's on your plate today.",
  "color": 16731212,
  "fields": [
    { "name": "Calendar", "value": "3 meetings, next at 10am", "inline": true },
    { "name": "Unread", "value": "5 emails, 2 flagged", "inline": true }
  ]
}
```

The `embeds.py` helper converts this to a `discord.Embed` object.

### Reactions

Pepper can react to messages using standard Discord emoji names. The channel server passes the emoji names through; the bot resolves them:

- Standard Unicode emoji: `thumbs_up`, `eyes`, `white_check_mark`, `fire`
- Custom server emoji: by name if available

### Typing Indicator

The bot shows a typing indicator when it POSTs a message to the channel server. The typing indicator automatically expires after 10 seconds in Discord. If Pepper takes longer, the bot can refresh it by tracking pending chat_ids and re-triggering typing until a reply arrives on the SSE stream.

### Error Handling

- Channel server unreachable: log error, send a brief Discord message ("I'm having trouble connecting right now")
- SSE stream disconnects: reconnect with exponential backoff (1s, 2s, 4s, max 30s)
- Discord API errors: log and continue, don't crash the bot

### Bot Setup Instructions

Documented steps for creating/configuring the Discord bot:

1. Go to Discord Developer Portal, create application
2. Bot section: create bot, copy token
3. Enable Message Content Intent under Privileged Gateway Intents
4. OAuth2 > URL Generator: select `bot` scope with permissions: Send Messages, Send Messages in Threads, Read Message History, Add Reactions, View Channels
5. Open generated URL to add bot to server
6. Save token to `integrations/discord/.env` as `DISCORD_BOT_TOKEN`

## Launch Script

### start-pepper (MVP)

```bash
#!/usr/bin/env bash
# Start Pepper's Claude Code session with the channel enabled
# Then start all integrations

PEPPER_DIR="$(cd "$(dirname "$0")/.." && pwd)"
PID_DIR="$PEPPER_DIR/.pids"
mkdir -p "$PID_DIR"

# Start Pepper (Claude Code with channel)
cd "$PEPPER_DIR"
claude --dangerously-load-development-channels server:pepper-channel &
echo $! > "$PID_DIR/pepper.pid"

# Wait for channel server to be ready
for i in $(seq 1 30); do
  curl -s http://localhost:8788/health > /dev/null 2>&1 && break
  sleep 1
done

# Start Discord bot
cd "$PEPPER_DIR/integrations/discord"
uv run python bot.py &
echo $! > "$PID_DIR/discord.pid"

echo "Pepper is running."
```

### stop-pepper (MVP)

```bash
#!/usr/bin/env bash
PEPPER_DIR="$(cd "$(dirname "$0")/.." && pwd)"
PID_DIR="$PEPPER_DIR/.pids"

for pidfile in "$PID_DIR"/*.pid; do
  kill "$(cat "$pidfile")" 2>/dev/null
  rm "$pidfile"
done

echo "Pepper stopped."
```

### Windows equivalents

`start-pepper.bat` and `stop-pepper.bat` with equivalent logic using `start /B` and `taskkill`.

## Configuration

All configuration via environment variables or `.env` files:

| Variable | Default | Description |
|----------|---------|-------------|
| `PEPPER_CHANNEL_PORT` | 8788 | Channel server port |
| `DISCORD_BOT_TOKEN` | (required) | Discord bot token |
| `PEPPER_ROUTE_TTL_HOURS` | 24 | How long to keep chat_id routing entries |

## Security

- Channel server binds to `127.0.0.1` only — not reachable from outside the machine
- Discord bot token stored in `.env`, gitignored
- No sender allowlist on the channel server for MVP (localhost-only is sufficient). When Pepper moves to her own box with remote access, add allowlist gating.

## Testing

- Channel server: unit tests for routing table, message validation, SSE streaming
- Discord bot: mock tests for message formatting, embed building, reaction handling
- Integration: start channel server, POST a message, verify it arrives on SSE stream with correct routing

## Out of Scope for MVP

- Permission relay (approve/deny tool use from Discord) — add later
- File attachments (images, documents from Discord) — add later
- Voice channels
- Slash commands (use natural language, not /commands)
- Multi-bot (only one Pepper instance)
- Remote access to channel server (localhost only for now)
